# SUI Move Contract Audit — 2026-06-04

In-session review of all 10 Move contracts in `contracts/sui/sources/`,
triggered by the operator report "withdrawal lower than expected for some
users" on 2026-06-03. Not a substitute for external review (Movebit /
OtterSec / Halborn) — this captures critical + high findings the
in-session pass surfaced + the fixes shipped.

## Severity scale

- **CRITICAL** — direct fund loss or contract bricking, exploitable today
- **HIGH** — fund-at-risk path or invariant violation requiring narrow conditions
- **MEDIUM** — degraded semantics, surprising behavior, future-scaling concern
- **LOW** — code quality, minor info disclosure

## Findings + fixes

### CRITICAL — community_pool_usdc.move share-math ignored off-chain assets

**Status:** fixed in commits `e8942f76` (oracle field + cron attestation)
and `252a4284` (open_hedge ratio rewire).

`calculate_assets_for_shares`, `calculate_shares_for_deposit`, and
`get_nav_per_share` used only `balance::value(&state.balance) +
VIRTUAL_ASSETS` as total assets — ignoring ~$44 of wBTC/wETH/SUI held on
the admin wallet and BlueFin collateral.

Live impact 2026-06-03: balance $0.41 vs true NAV $44.99. A member
withdrawing 3 shares (10% of pool) would receive $0.135 instead of $4.50.
Symmetric over-issuance on deposit.

Fix: added external NAV oracle via dynamic_field (upgrade-safe — no
struct field added). `admin_attest_external_nav(AdminCap, value, clock)`
with 30% per-call change cap. `admin_set_external_nav_required(bool)`
gates strict-mode freshness assert. Math functions use new internal
`total_assets_including_external`. Cron pushes external NAV every tick
via `attestExternalNav()` in sui-community-pool/route.ts.

### CRITICAL — community_pool_usdc.move close_hedge accepted unverified PnL

**Status:** fixed in commit `252a4284`.

Original `close_hedge` took `pnl_usdc + is_profit + funds` as inputs and
joined whatever funds it got into the pool, regardless of consistency.
A compromised AgentCap holder could call with `is_profit=false,
pnl_usdc=collateral, funds=Coin::zero()`, removing the hedge from
active_hedges and decrementing total_hedged_value without returning any
USDC. Silent drain proportional to any open hedge on-chain.

Fix: assert `coin::value(&funds) >= expected_return` where expected =
`is_profit ? collateral + pnl : max(0, collateral - pnl)`. Generous
returns (e.g. funding rewards captured during hedge life) still accepted;
excess goes into pool balance.

### CRITICAL — community_pool.move close_pool_hedge same drain vector

**Status:** fixed in commit (this commit).

Same shape of bug in the SUI-native sibling. CLAUDE.md flags the USDC
pool as the production target, but if the SUI-native pool is still
hot-callable, the AgentCap drain works identically.

Fix: same funds-verify assert added to `close_pool_hedge`.

### HIGH — community_pool_usdc.move open_hedge ratios broke after external-NAV fix

**Status:** fixed in commit `252a4284`.

Yesterday's `get_total_nav` change to include external_nav cascaded into
`open_hedge`'s reserve-ratio, max-hedge-ratio, and daily-hedge-cap
checks. At $0.41 on-chain vs $44 external, the 20% reserve floor became
$8.89 — unreachable on-chain. Every cron auto-hedge tick would have
reverted with `E_RESERVE_RATIO_BREACHED`.

Fix: new `get_onchain_nav()` = `balance + hedge_state.total_hedged_value`
used by the open_hedge ratio checks (which are about the contract's
controllable USDC, not depositor wealth). Share-pricing math still uses
the full `get_total_nav()`.

### HIGH — community_pool_usdc.move emergency_withdraw FCFS race

**Status:** fixed in commit `252a4284`.

When the pool is paused or circuit-tripped, members can call
`emergency_withdraw`. Original logic capped payout at
`min(fair_share_value, available_balance)`. First member to call could
drain the balance up to their full fair share, leaving later members
with 0. Severity increased after the external-NAV fix because fair share
value now often exceeds available balance.

Fix: pro-rata cap. `amount = min(fair_share, available × shares /
total_shares)`. Every member gets a proportional cut regardless of call
order. Defensive guard on `amount > 0` around the transfer.

### CRITICAL — hedge_executor.move oracle manipulation by AdminCap

**Status:** fixed in commit (this commit). Contract is testnet-only today
per the deploy commit (`c2205326`).

`update_price_feed` accepted any positive price under AdminCap. No
bounded change, no timestamp, no anti-manipulation. Admin could push
arbitrary price (e.g. drop BTC to $1) between user transactions, causing
`close_hedge` to compute massive PnL either direction and drain the
collateral pool. Single-block rug.

Fix: bound per-update change to 50% of prior value. Still permissive
enough for legitimate volatility (BTC's worst day in 2020 was -45%) but
stops the single-block rugpull. Long-term move to Pyth oracle integration.

### HIGH — community_pool.move shares math ignores hedge state

**Status:** NOT FIXED. Flagged for the same upgrade window as
community_pool_usdc.

community_pool.move (SUI-native, deprecated per CLAUDE.md but possibly
still hot-callable) has the same balance-only share math. `get_total_nav`
there only returns `balance::value(&state.balance)` — even less than
the USDC pool's pre-fix version, which at least included
`hedge_state.total_hedged_value`. If SUI native pool has any off-chain
positions, withdrawing members are underpaid.

Recommended fix: same external-NAV oracle pattern applied to USDC pool.
~150 lines. Defer until/unless the SUI-native pool is confirmed to have
real members + off-chain positions; the USDC pool is the live target.

## Findings observed but not patched in this pass

### MEDIUM — fee math overflow above $500M NAV

`collect_management_fee_internal` computes `(nav * fee_bps *
time_elapsed_sec) / (BPS_DENOMINATOR * SECONDS_PER_YEAR)`. At $500M NAV
× 50 bps × 1 day = 2.16e18, within u64 (1.8e19) but tight. A week of
skipped fee collection at $500M = overflow.

Already mitigated by `NAV_SAFETY_CEILING_USDC` halt and the daily
`sui-collect-fees` cron heartbeat. Long-term: u128 math redeploy
(T4-B in the pre-deposit TODO).

### MEDIUM — performance fee can be inflated via attested external NAV

After the 2026-06-03 oracle fix, the admin-attested external NAV feeds
into `get_nav_per_share`, which feeds into `collect_performance_fee_internal`'s
high-water-mark check. An admin who slightly inflates external NAV
attestation would trigger HWM crossings + performance fee accrual.

Mitigations in place: 30% per-call change cap, fees go to treasury (also
admin-controlled), so this is transparency rather than theft. Long-term:
multi-attestor consensus or Pyth-backed valuation oracle.

### MEDIUM — open_hedge / close_hedge race window

The cron calls `open_hedge` then off-chain swaps USDC to wBTC. Between
the on-chain tx and the off-chain settlement, NAV is misreported as
"hedge_state value moved to admin" while no real off-chain asset exists
yet. Window is ~1-5 seconds in normal operation but could be exploited
by a sandwicher in pathological conditions.

Mitigation: cron operates with sub-second cadence on the swap leg; pool
caps individual hedge size; this is acceptable for current volumes.

### LOW — admin_reset_hedge_state emits UsdcPoolPaused event for non-pause action

`admin_reset_hedge_state` and `admin_reset_daily_hedge` both emit
`UsdcPoolPaused { paused: false }` events as "signal state change"
proxies. Indexers and event subscribers parsing pause events will see
false positives. Recommend dedicated event types.

### LOW — emergency_withdraw doesn't decrement member_count

After full withdrawal in `emergency_withdraw`, `member.shares` is set
to 0 but the member row stays in the table and `member_count` is not
decremented. `get_pool_stats` will report stale member count after an
emergency drain. Cosmetic — does not affect correctness.

## Phase 2 — remaining contracts (audited 2026-06-04 later same day)

### CRITICAL — zk_proxy_vault.move execute_withdrawal cross-proxy drain

**Status:** fixed in commit (this commit).

`execute_withdrawal` asserted `pending.owner == sender` but never that
the `PendingWithdrawal` actually belonged to the `ProxyBinding` passed
in. On SUI, both objects are shared and can be passed by any caller. So
an attacker could:

  1. Create their own proxy A, deposit a small amount, request a
     time-locked withdrawal of $X.
  2. Wait the time-lock (24h default).
  3. Call execute_withdrawal with `pending=their_pending(A, X)` and
     `proxy=victim_shared_proxy_B`. The function decrements `state.tvl`
     by X, marks the pending executed, and splits X SUI from B's
     balance to send to `pending.owner` (= attacker).

End result: $X drained directly out of any victim proxy with sufficient
balance. Bound only by the attacker's initial reservation + the
time-lock wait.

Fix: assert `pending.proxy_id == object::id(proxy)`. Single line; binds
each pending to its source proxy.

### CRITICAL — zk_proxy_vault.move cancel paths inflate wrong proxy accounting

**Status:** fixed in commit (this commit).

Same shape in `cancel_withdrawal` and `guardian_cancel_withdrawal`: no
check that the pending refers to the proxy passed in. After cancelling
the pending, the function credits `proxy.deposited_amount += pending.amount`
to whatever proxy was passed — inflating B's bookkeeping while A's real
balance stays decremented. Sequentially cancellable to manufacture
arbitrary fake credit on any owner's proxy.

Fix: same `pending.proxy_id == object::id(proxy)` assert on both cancel
paths.

### HIGH — community_pool_timelock.move any proposer can cancel any operation

**Status:** fixed in commit (this commit).

`cancel_operation` required only a `ProposerCap`. In a multi-proposer
multisig setup (the entire raison d'être of the timelock), any single
proposer could permanently cancel operations queued by the others —
turning the N-of-N safety into a 1-of-N veto. A rogue or compromised
proposer can paralyse the timelock.

Fix: assert `operation.proposer == ctx.sender()`. Proposers can now only
cancel their own operations. Cross-proposer cancellation must go through
the admin path or by scheduling a counter-operation.

### MEDIUM — community_pool_timelock.move pause didn't halt single-op execution

**Status:** fixed in commit (this commit).

`schedule_operation`, `schedule_batch`, and `execute_batch` all checked
`!paused`. `execute_operation` (the single-op path) did not. So when
admin paused for an active incident, queued ops could still be drained
through the single path.

Fix: added `assert!(!state.paused, E_PAUSED)` at the top of
execute_operation. Now pause is complete.

### HIGH — `verify_zk_proof` is a stub across all ZK paths

**Status:** documented; not fixed in this pass.

Every "ZK proof verification" in the SUI Move side is a length check:

  - `zk_proxy_vault::verify_zk_proof` — returns true if proof ≥64 bytes and
    public_inputs.length ≥ 4 (zk_proxy_vault.move:601-629)
  - `zk_hedge_commitment::verify_settlement_proof` — returns true if
    proof ≥ 64 bytes and `commitment_hash.length > 0`
    (zk_hedge_commitment.move:430-439)
  - `zk_verifier::execute_commitment` — only asserts
    `hash(proof_data).length > 0`, which is always true for non-empty
    input (zk_verifier.move:267-295)
  - `rwa_manager::execute_strategy` — `verified = zk_proof_hash.length > 0`,
    emits event but doesn't gate anything (rwa_manager.move:373-406)

This means the "ZK verification" branding is theatre everywhere on the
SUI side. The real STARK verification happens in the Python prover
(`zkp/api/server.py`); these Move contracts only check that *something*
was submitted. Practical impact varies by call site:

  - In `zk_proxy_vault::withdraw`, the proxy.owner check (line 388) is
    the actual auth gate; ZK proof bypass means "anyone with the proxy
    owner key can withdraw" which is identical to the proxy.owner check
    alone — proof acts as ceremony but adds no security.
  - In `zk_hedge_commitment::settle_commitment`, the RelayerCap gate
    is the real auth; proof has no marginal security.
  - In `zk_verifier::execute_commitment`, the VerifierCap gate is the
    real auth.

Recommended: either implement real on-chain verification (groth16-style
verifier, Pyth-attested proof root, or commitment-to-prover-output
pattern with a centralized prover signing the result) — or remove the
proof input entirely and stop calling these contracts "ZK".

Defer until external audit defines the right pattern.

### MEDIUM — rwa_manager.move portfolio rebalance/strategy are no-ops

**Status:** documented; no fix needed.

`rebalance` updates `last_rebalance` and emits an event but performs no
actual rebalancing. `execute_strategy` emits an event but does no
strategy execution. These are demo stubs. The real RWA accounting must
live elsewhere or be reimplemented. Flagged for product clarity, not a
security issue.

### MEDIUM — payment_router.move sponsor fund semantics

**Status:** documented; no fix needed.

`deposit_sponsor_fund` accepts SUI from anyone. `withdraw_sponsor_fund`
is admin-gated and sends to `fee_recipient` (the admin's beneficiary),
not back to any sponsor. The contract is effectively a donation pot,
not a sponsor reimbursement system as the naming suggests. Either:
  - Rename to make the donation semantics explicit
  - Or change withdraw to a sponsor-specific recipient

Also: `record_sponsored_tx` is accounting-only — it increments
counters but moves no SUI. The contract doesn't actually pay out
sponsored gas; it just tracks intent. Document or wire up payment.

### MEDIUM — bluefin_bridge.move is a pure off-chain mirror

**Status:** documented; no fix needed.

The bridge holds no funds — `BluefinPosition` objects track open/close
state mirrored from BlueFin's off-chain DEX by a RelayerCap. There's no
fund movement path on-chain, so the worst-case attack is a malicious
relayer producing fake positions or sync data, which would corrupt
analytics but cannot move funds. Mitigation lives in the relayer
authorization (RelayerCap distribution), not in the contract logic.

### MEDIUM — bluefin_bridge.move position transferred to trader, not shared

**Status:** documented; no fix needed.

`record_position_open` uses `transfer::transfer(position, trader)` which
makes the position owned by the trader. `record_position_close` and
`sync_position` take `&mut BluefinPosition` — meaning the relayer needs
the trader to wrap the object in their own tx to update it. In practice
this means the relayer cannot sync without trader cooperation, which
defeats the relayer model. Either share the object or accept that sync
will lag.

### LOW — zk_hedge_commitment.move pending_commitments O(n²) drain pattern

**Status:** documented; no fix needed.

`create_batch` repeatedly calls `vector::remove(&mut pending_commitments, 0)`
which is O(n) per call, O(n²) total over `batch_size` elements. At
MAX_BATCH_SIZE=100 this is 10k operations — fine. But at any larger
size, performance degrades sharply. Don't raise MAX_BATCH_SIZE without
refactoring to swap_remove or a queue structure.

## Phase 3 — remaining "documented" findings fixed (2026-06-04 evening)

### HIGH — ZK is theatre — RESOLVED via opt-in ed25519 prover attestation

**Status:** fixed in commit (this commit). Applied to all 4 affected
contracts: `zk_verifier`, `zk_proxy_vault`, `zk_hedge_commitment`,
`rwa_manager`.

Real on-chain STARK verification in Move is not practical today (no
pairing primitives, gas would dwarf any realistic trade size). The
realistic upgrade path is to make the off-chain STARK prover an
ed25519-signed attestor — equivalent in trust model to Pyth, but
cryptographically anchored on-chain instead of being a length check.

Pattern applied to all four contracts:

  1. New entry function `admin_set_prover_pubkey(AdminCap, state,
     pubkey: vector<u8>)`. Pubkey must be exactly 32 bytes (ed25519
     public key). Empty vector clears the field for emergency
     reversion to insecure mode.
  2. Storage uses `sui::dynamic_field` on the state object's UID with
     a distinct key per contract (`zkv_prover_pubkey_v1`,
     `zpv_prover_pubkey_v1`, `zhc_prover_pubkey_v1`,
     `rwa_prover_pubkey_v1`). This avoids any struct migration in
     already-deployed package state.
  3. Verification functions now check `df::exists_(state.id, KEY)`.
     If a pubkey is set, the first 64 bytes of the submitted proof
     must be a valid `sui::ed25519::ed25519_verify` signature over a
     context-appropriate message:
       - `zk_verifier::verify_proof[_for_portfolio]`: signs the
         `commitment_hash`
       - `zk_verifier::execute_commitment`: signs the
         `commitment.commitment_hash`
       - `zk_proxy_vault::verify_zk_proof`: signs the derived
         `binding_hash(owner, proxy_address)`
       - `zk_hedge_commitment::verify_settlement_proof`: signs the
         `commitment_hash`
       - `rwa_manager::execute_strategy`: signs
         `bcs(portfolio_id) || strategy_bytes`
  4. If no pubkey is set, the legacy "non-empty / ≥64 bytes" check is
     preserved. This is the opt-in migration path — operators MUST
     set the pubkey to remove the theatre. The new `has_prover_pubkey`
     view function lets off-chain ops detect insecure mode.

This converts the contracts from "ZK proof = byte length check" to
"ZK proof = cryptographic attestation from a known prover key" the
moment an operator runs `admin_set_prover_pubkey`. The trust model
becomes equivalent to Pyth: trust the prover key, not the contract
internals.

External audit (T4-C) should validate the message-binding for each
contract and consider whether the signed context covers enough state
to prevent replay across contracts or sessions.

### MEDIUM — payment_router sponsor fund semantics — RESOLVED

**Status:** fixed in commit (this commit).

Two changes:

  1. `withdraw_sponsor_fund` now takes an explicit `recipient: address`
     argument. Previously the destination was hardcoded to
     `state.fee_recipient`, which silently funnelled "sponsor fund"
     deposits to the admin's beneficiary regardless of intent. Admin
     now has to declare the destination explicitly per call.
  2. New entry function `pay_sponsor_gas(SponsorCap, state, beneficiary,
     gas_amount, clock)` actually transfers SUI from `state.sponsor_fund`
     to the beneficiary, enforcing the sponsor's daily limit. This is
     the real payment path. `record_sponsored_tx` remains for
     bookkeeping-only flows (e.g. when gas was sponsored via the SUI
     tx-level sponsored-transaction mechanism and only the daily-limit
     accounting needs to be recorded).

### MEDIUM — bluefin_bridge relayer sync impossibility — RESOLVED

**Status:** fixed in commit (this commit).

`record_position_open` now calls `transfer::share_object(position)`
instead of `transfer::transfer(position, trader)`. The relayer can now
take `&mut` on the position from any transaction to call
`sync_position` and `record_position_close` without needing the trader
to wrap the object first. Trader audit trail preserved via the
`position.trader` field.

### MEDIUM — rwa_manager strategy stubs — RESOLVED via real verification

**Status:** the underlying "no real strategy logic" is documentation —
real strategy actions live off-chain in the orchestrator. The audit
concern was that the on-chain `execute_strategy` function emitted a
"verified" event under a fake length check. With phase 3's ed25519
attestation wired into `execute_strategy`, the function now requires a
real prover signature over `portfolio_id || strategy` bytes to emit the
event. Strategy events from this contract are now cryptographically
attested, even though the on-chain action itself remains a no-op.

### LOW — zk_hedge_commitment O(n²) batch drain

**Status:** documented (not patched). Tried an O(n) rewrite and reverted
— at MAX_BATCH_SIZE=100 the cost is ~5000 ops, well within gas, and the
correct O(1)-amortised refactor needs a head-index queue structure that
should be designed alongside the external audit. Added an in-code
warning not to raise MAX_BATCH_SIZE above ~200 without that refactor.

## External review still required

All 10 contracts have been touched at least once by this in-session
pass, but in-session ≠ external. T4-C remains open. Recommend the
external auditor focus on:

1. **The "ZK is theatre" finding (HIGH)** — does the system need real
   on-chain ZK verification, and if so what pattern (groth16 verifier,
   committee-signed prover output, Pyth attestation)?
2. **community_pool_usdc share math under high-frequency external NAV
   attestation** — the 30% per-call change cap was deliberately loose
   for early adoption; auditor should advise on tightening.
3. **AdminCap concentration risk** across all contracts — every contract
   has an AdminCap with broad powers; multisig migration (T4-A) is the
   right answer but the auditor should validate the migration plan.
4. **Reentrancy + flash-loan composability** — none reviewed in this
   pass.

## Recommended next steps before any meaningful AUM

1. **Don't unpause the pool until the audit fixes ship.** The fixes in
   this doc + the original 2026-06-03 oracle fix are all in the same
   upgrade payload — apply them together.

2. **Engage external auditor for the 7 unreviewed contracts.** Budget
   $30-100k, 4-8 weeks (already T4-C in the pre-deposit TODO).

3. **Move `AdminCap` to multi-sig** (already T4-A). Reduces the impact
   of every "compromised AdminCap holder" finding in this audit by
   adding a quorum requirement.

4. **Replace admin-attested oracles with Pyth** wherever they exist.
   Reduces trust surface from "admin can rugpull via stale/false price"
   to "Pyth must remain functional."

5. **Set up Movebit / OtterSec / Halborn engagement** for ongoing review.
